Pipeline at a Glance

This pipeline has one judgment step that everything depends on — AI-guided capture of business context into a Service Understanding Document (SUD). The six stages after it are mechanical transforms. If the capture is wrong, every downstream artifact is wrong. The SUD is the durable asset; dashboards are disposable projections regenerated from it.

Three things to know
1. The question instrument (50+ methodology questions) is the highest-leverage component — changing a question changes all future SUDs.
2. The most dangerous failure is content quality — a SUD can pass all validators but contain wrong understanding. Only human review catches this.
3. Three of five promised factory outputs (alerts, playbooks, stories) are designed but not built.
How it works
The pipeline exists because manual dashboard creation doesn't scale, and business observability requires structured understanding before dashboards can be meaningful.

Four paths through the system — capture, profile management, deployment, and regression. The SUD YAML is the single durable asset; everything downstream is regenerable from it.

Key Flows

  • Capture path: Source material → bos-onboard → SUD YAML (the durable asset; everything downstream is regenerable)
  • Profile path: SUD YAML → ingest_sud.py → BOS app (profile becomes queryable and manageable)
  • Deployment path: BOS app → ServiceExportDto → dashboard factory → Grafana
  • Regression path: Golden fixture → validators → factory tests

What can go wrong
Most failures are caught by validators and produce structured errors. The dangerous one — wrong but schema-valid understanding — can only be caught by human review.
Silent failure risk
The most dangerous failure is SUD content quality — a SUD can be schema-valid but contain shallow, ungrounded understanding. Provenance discipline and quality checks mitigate, but can't catch wrong understanding, only unsupported understanding.

Why it's designed this way
Two reinforcing loops drive improvement (more onboardings → better quality). Two balancing loops prevent degradation (validation gates, factory contracts). The delay in schema promotion (weeks to months) is the bottleneck in the reinforcing loop.
Reinforcing — SUD as growth sensor
Each service onboarded surfaces schema-ahead fields → accumulates evidence for data model evolution → richer schema → richer SUDs → richer factory output. Multiple SUDs asking the same question is the signal to promote a field.
Reinforcing — Golden fixture accumulation
Each completed SUD becomes a fixture → more exemplars → better pattern matching → faster, higher-quality onboarding. Currently 10+ fixtures across Astronomy Shop and Pitstop domains.
Balancing — Validation gates
Validator strictness constrains quality floor → prevents degradation → but creates friction during capture. Balance point: strict enough to prevent garbage, loose enough for iteration.
Balancing — Factory acceptance contract
validate_onboarding.py tests ensure passing = factory will consume without error. Couples validation to factory's actual requirements, preventing drift in either direction.

Delay Characteristics

  • Schema-ahead → promotion: Weeks to months (requires multiple SUDs showing same gap)
  • Source quality → SUD quality: Immediate, but only visible after render + human review
  • Factory improvement → dashboard improvement: Zero delay for new services; existing dashboards require re-generation
The highest-leverage interventions are at the paradigm and rules level — the organizing model and the question instrument. Most active work targets these levels, not parameters.
Paradigm — "Understanding first, artifacts second"
SUD as intermediate representation, dashboards as disposable projections. If this inverts, the entire system degrades.
Goal — Four-layer model
Business Health → Business Impact → Process → System. Organizes signals, dashboard layout, and stakeholder communication.
Rules — Question instrument (sud-questions.yaml)
Determines WHAT gets captured. Adding/removing/reframing a question changes all future SUDs.
Information flow — Provenance discipline
Extracted / inferred / constructed labeling. Makes understanding quality visible to reviewers.
Feedback — Schema-ahead → schema promotion loop
Each promotion unlocks new factory outputs for ALL future services.
Stock — The SUD YAML itself
The durable asset. Everything downstream regenerates from it.
Flow — Adding more source material
Improves SUD quality for one service.
Parameter — Validation thresholds
Tweaking word count, citation count. Adjusts quality floor marginally.
Current interventions at the right levels
The question instrument (Very High) and schema evolution (High) are active work areas. The paradigm (Highest) is stable and proven.
SUD YAML as intermediate representation
Alternatives: direct CSV templates, direct database entry, JSON-first
YAML is human-readable + AI-writable; decouples judgment (capture) from mechanics (factories); the durable asset survives factory changes.
One judgment step, six mechanical transforms
Alternatives: all-AI pipeline, all-human pipeline
Judgment can't be automated reliably (understanding requires inference); mechanics should be automated (transforms are deterministic).
Question instrument as methodology
Alternatives: freeform capture, checklist, template
Questions with IDs enable traceability to schema fields, enhanced guidance, prior-art references, downstream population mappings.
Discriminated union on signal_type
Alternatives: single signal shape with optional fields
sli_ratio (numerator/denominator) and sli_threshold (numerator/operator/value) are structurally different; optional fields allow invalid combinations.
Datasource-agnostic factory
Alternatives: per-datasource generators
Same dashboard structure regardless of telemetry backend; query_adapters.py handles syntax; datasource-config.yaml maps to Grafana UIDs.
Provenance labeling (extracted / inferred / constructed)
Alternatives: no provenance, binary confident/unconfident
Three-level maps to audit trail: extracted = verifiable, inferred = challengeable, constructed = flagged for validation.
Atomic import (create-or-replace)
Alternatives: incremental updates, merge semantics
Prevents partial states; safe re-onboarding; simpler error recovery.
Golden fixture as structural exemplar
Alternatives: schema-only, multiple exemplar formats
Single ~600-line exemplar provides field naming, nesting order, block scalar patterns — prevents drift across SUDs.
Four-layer dashboard organization
Alternatives: alphabetical, by datasource, by signal type
Business Health at top → System Health at bottom maps to stakeholder reading order.

Well-designed workflow with one primary refinement needed: the PO should review the rendered HTML before deployment, but the pipeline doesn't require it.

Actors: PE (runs pipeline, reviews, deploys), bos-onboard skill (judgment step), BOS app (stores profiles), Service owner/PO (provides sources, consumes dashboards).

CriterionAssessment
Goal-Process AlignmentAligned — SUD captures understanding; factory manifests it.
Step NecessityAll 7 stages necessary. One debatable: Export (Stage 5) — factory could read directly, but API round-trip provides contract enforcement.
Step SufficiencyGap — No automated regression test post-deploy. No provenance quality trending across SUDs.
Sequence OptimalityOptimal. Each stage gates the next. One minor optimization: Render could run parallel with Load.
Actor AppropriatenessGap — PO doesn't interact with pipeline; PE proxies. PO should review rendered HTML before deployment.
Failure RecoveryGraceful at each stage. One gap: successful import but failed deploy leaves profile without dashboard.
Verdict: Well-designed
Primary refinement: enforce PO review gate between render and deploy. The pipeline produces the artifact for this review but doesn't require it.

Expert Redesign Recommendations

  1. Add PO review gate between render and load
  2. Add post-deploy smoke test (HTTP 200 with data?)
  3. Make export → factory a single step for local runs
  4. Add telemetry on SUD quality trending (provenance distribution over time)
